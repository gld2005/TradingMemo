# 技术边界

## 核心原则

1. 本项目优先本地优先；
2. 不依赖云端才能运行；
3. 第一版不接 AI API；
4. 第一版不接行情 API；
5. 第一版不接券商 API；
6. 第一版不做登录系统；
7. 图片保存到本地文件夹；
8. 数据保存到本地数据库或本地结构化文件；
9. 后续技术选型必须服务于轻量 Windows 桌面浮窗；
10. 不允许为了炫技引入复杂架构。

## 选型约束

- 具体桌面框架和本地存储方案留待对应 Batch 在上述边界内确认，Batch 0 不做过早实现。
- 新依赖必须有明确的第一版需求依据，并以体积、启动速度、维护成本和 Windows 兼容性为主要判断标准。
- 应用的核心记录、浏览、搜索、导出和备份能力在离线环境下必须可用。

## Batch 1 已确认技术路线

- 桌面框架采用 Electron，渲染层采用 React + TypeScript + Vite。
- 原计划优先评估 Tauri，但 Batch 1 执行环境缺少 Rust 与 Cargo，无法初始化和验证；因此按 Batch 1 授权的降级规则使用 Electron。
- 当前只建立主窗口、浮窗占位窗口、页面占位、基础组件和明暗主题，不引入状态管理库、大型 UI 组件库、数据库或外部服务依赖。
- 后续 Batch 应沿用本文件的本地优先和范围禁区，不得因框架选择增加产品边界。

## Batch 2 窗口能力

- 使用 Electron 官方 `BrowserWindow` 实现单个无边框置顶浮窗，使用 CSS `-webkit-app-region` 划分拖动和交互区域。
- 使用 Electron 官方 `globalShortcut` 注册唯一快捷键 `Alt + J`；注册失败时保留主窗口显示/隐藏入口作为降级方式。
- 使用 `preload`、`contextBridge` 和最小 IPC 通道连接渲染层与主进程，保持 `contextIsolation` 开启且不向页面开放 Node.js。
- 展开/迷你状态、输入文字和静态分类均只保存在进程内存中，不写入数据库、文件或浏览器持久化存储。

## Batch 3 本地数据能力

- 使用 Electron `app.getPath("userData")` 定位用户数据目录，并在其 `app-data/notes.json` 中保存结构化 JSON；不把真实用户数据写入源码目录。
- Electron 主进程使用 Node.js 文件系统实现 `StorageService`，渲染进程仅通过 `contextBridge` 暴露的最小 IPC 调用访问数据，不开放 Node.js 权限。
- `NotesRepository` 提供初始化、读取笔记、新增纯文字笔记、读取分类和初始化默认分类能力；数据文件保留 `schemaVersion: 1`，不引入 ORM 或复杂迁移系统。
- 数据文件缺失或为空时自动初始化；文件损坏、读取失败和保存失败时向界面提供友好错误反馈，不依赖云端错误上报。
- Attachment 在本批次只有数据结构，不实现图片粘贴、拖入、预览或真实附件流程。

## Batch 4 图文记录能力

- 浮窗通过浏览器标准粘贴/拖放事件维护图片草稿，保存前不创建正式 Note 或 Attachment；支持 PNG、JPG、JPEG、WebP，单张最大 10MB，单条笔记最多 10 张。
- 渲染进程通过 `contextBridge` 传递图片字节；Electron 主进程将文件写入 `app.getPath("userData")/app-data/attachments/YYYY-MM/note_<id>/`，文件名使用生成 ID，不依赖原始文件名。
- Note 的 `attachmentIds` 与 Attachment 的 `noteId` 建立双向可核对关联；Note 和 Attachment 记录在一次结构化数据写入中提交，失败时回滚本次附件目录。
- `notes.json` 使用同目录临时文件写入后替换正式文件，避免失败覆盖破坏已有记录。
- 今日记录只按已登记 Attachment ID 通过最小 IPC 读取附件字节，不向渲染层开放 Node.js 或任意文件路径读取能力；本地文件丢失时显示占位状态。
- 本批次只提供前 3 张缩略图、`+N` 和只读大图 overlay，不实现编辑、删除、独立图片库、OCR、内容识别、搜索或分类标签管理。

## Batch 5 分类与标签能力

- 沿用 `NotesRepository` 和 `app-data/notes.json`，不引入数据库、ORM、状态管理库或大型 UI 组件库。
- 数据文件升级为 `schemaVersion: 2`；读取 v1 数据时为旧 Note 补齐空的 `categoryId`、`tagIds`、`stockName`、`stockCode` 和缺失的 `attachmentIds`，保留 Attachment 数据与已有文件关联。
- Category 与 Tag 的新增、重命名、删除、名称去空格、重复校验和引用保护统一在 Electron 主进程 Repository 中处理，渲染层只通过最小 IPC 调用。
- 默认分类通过轻量初始化标记只在首次建立数据时创建；分类按 `sortOrder` 展示。分类或标签被 Note 引用时禁止删除，名称修改后通过 ID 关联自动显示新名称。
- Tag `usageCount` 根据 Note 的 `tagIds` 动态计算，避免持久化计数漂移；单条 Note 最多关联 10 个标签。
- 浮窗只提供分类选择、已有/新标签选择及股票名称/代码文本输入，不提供分类管理、搜索、推荐、行情查询或自动补全。
- 今日记录只增加分类、标签和股票文本的最小展示，不实现知识库、详情页、编辑、删除、全文搜索或组合筛选。

## Batch 6 主窗口知识库能力

- 沿用 Electron + React + TypeScript、`NotesRepository`、`app-data/notes.json` 和现有附件目录，不新增数据库、状态管理库、UI 组件库或外部服务。
- 知识库通过现有最小 IPC 并行读取 Note、Category、Tag，再按 Note 的 `attachmentIds` 读取 Attachment 记录；分类数量与三栏展示均在渲染层由本地数据派生。
- Repository 新增单条笔记更新与删除能力。更新会验证分类、标签和附件数量，先写入新增图片与结构化数据，结构化写入失败时清理本次新增文件且不改变旧记录。
- 移除图片或删除笔记时，先从 Note 与 Attachment 结构化记录中解除关联并安全写入，再逐个清理已登记的本地文件；文件清理失败通过警告返回，不恢复已成功提交的数据，也不留下损坏引用。
- 更新与删除成功后继续使用 `notes-changed` 广播刷新主窗口和浮窗相关本地视图；不引入复杂全局状态。
- 本批只允许“全部笔记 / 未分类 / 单分类”浏览，不实现全文或关键词搜索、标签/股票/日期筛选、组合筛选、导出、备份或恢复。

## Batch 7 本地搜索与筛选能力

- 沿用 Electron + React + TypeScript、现有安全桥接和本地结构化数据，不新增依赖、状态管理库、搜索引擎、IPC 通道或远程服务。
- 搜索与筛选在渲染层对已加载的 Note、Category 和 Tag 数组进行纯函数过滤；不读取任意文件，不上传笔记内容，不建立全文索引或分词系统。
- 关键词使用去除首尾空格后的大小写无关字符串包含匹配，覆盖标题、正文、股票名称/代码、分类名称和标签名称。
- 分类、标签、股票、创建日期、是否带图片和关键词按 AND 组合；多个标签也使用 AND。日期快捷范围按本地时间计算，自定义日期包含起止日。
- 左侧分类浏览和筛选面板共享单一分类状态。过滤结果变化时保留仍可见的当前笔记，否则选择首条或清空详情；编辑、删除和 `notes-changed` 刷新后重新基于当前条件派生结果。
- 本批不实现搜索历史、搜索建议、自动补全、模糊搜索库、相关性排序、保存筛选方案、导出、备份、恢复或 Batch 8 能力。

## 技术禁区

第一版明确禁止 AI 自动整理或每日复盘、行情接入、荐股、券商同步、自动交易、独立图片库、人工总结模块。因此不得接入 AI API、行情 API、券商 API，也不得为这些能力创建适配层、预留接口或数据模型。

## Batch 8 设置、导出与打包边界

- 设置保存于 `app-data/settings.json`，文件系统、对话框、快捷键和恢复只在 Electron 主进程处理；
- 渲染进程仅使用最小 `contextBridge` API，不获得 Node.js 或任意路径访问能力；
- 导出、备份和恢复均为本地文件操作，文件夹式备份不使用压缩、云端或远程服务；
- 恢复先校验并创建安全备份，覆盖失败时回滚；
- Windows 打包采用 electron-builder x64 NSIS 离线目标，无自动更新、遥测或发布上传。
